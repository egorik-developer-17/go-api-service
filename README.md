# go-api-service

`go-api-service` — учебный API-сервис на языке Go, используемый в качестве практической базы для выпускной квалификационной работы на тему **«Автоматизация развертывания проекта и реализация CI/CD»**.

Проект демонстрирует полный цикл доставки изменений: от внесения кода в GitHub-репозиторий до автоматической проверки, сборки Docker-образа, публикации образа в GitHub Container Registry и развертывания сервиса на VPS-сервере через Docker Compose.

## Связь с дипломной работой

В рамках дипломной работы проект `go-api-service` используется для практической реализации CI/CD-конвейера. В аналитической части диплома проект рассматривается как API-сервис на Go, для которого необходимо автоматизировать процессы проверки, сборки, тестирования, контейнеризации и развертывания в тестовую среду на VPS.

Изначально CI/CD-контур проекта включал базовые этапы: проверку кода, запуск тестов, сборку приложения, проверку Docker-образа и деплой на VPS. При этом исходная схема доставки предполагала передачу архива исходного кода на сервер и сборку контейнера непосредственно на VPS. В расчетно-практической части диплома данная схема совершенствуется: Docker-образ собирается и проверяется в GitHub Actions, публикуется в GitHub Container Registry, а VPS-сервер загружает уже готовый образ и запускает его через Docker Compose.

Таким образом, репозиторий является практической реализацией положений дипломной работы и демонстрирует переход от базовой автоматизации развертывания к более воспроизводимой CI/CD-схеме на основе готового контейнерного артефакта.

## Основные возможности

- REST API для работы с товарами.
- HTTP endpoint `/health` для проверки состояния сервиса.
- Локальный запуск приложения на Go.
- Запуск через Docker Compose.
- Многоэтапная сборка Docker-образа.
- Автоматические тесты, включая HTTP-тестирование обработчиков.
- CI/CD-конвейер на GitHub Actions.
- Публикация Docker-образа в GitHub Container Registry.
- Автоматическое развертывание на тестовый VPS.
- Проверка работоспособности после деплоя.
- Возможность восстановления сервиса через Docker Compose или повторный запуск CI/CD workflow.
